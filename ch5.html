The Mobile App

Field worker experience. How offline truly works. Dirty vs clean records. The SDK engine that renders every form dynamically.

🔑
True Offline Capability

Fill forms and save with zero internet. Data stores locally on the device and syncs to the server when connectivity returns. No data loss. No time limit.

App Architecture

🏠
App Shell

Lightweight outer layer. Provides branding, project config, and client-specific form handlers. Different per project.

⚙️
SDK Engine

The heavy lifter: reads DocType definitions, renders forms as native screens, stores data locally (SQLite), syncs with server, manages auth tokens. Shared across all projects.

💡
No form is hardcoded in the app

Every form is built dynamically from the server's DocType definition. Form changes on the server update the app automatically on next sync.

Login — Step by Step

1

App checks if mobile service is enabled

Quick server check, no credentials needed. Shows message and stops if disabled.

2

User enters credentials

Username/password or mobile + OTP. On success: auth tokens (30-day validity), enabled forms list, permissions, language preference.

3

Initial sync downloads everything needed

All form definitions, existing records, and translations for the user's language.

4

App is ready — internet can be switched off

All data for offline work is stored locally. The user is fully self-sufficient.

Online vs Offline Save

  1. Data sent to server immediately
  2. Server runs all Python validation scripts
  3. If passes → server assigns permanent ID, record becomes Clean
  4. If fails → error shown to user, nothing saved
  1. Data saved to local database on the device
  2. App assigns a temporary local ID
  3. Record marked Dirty — needs to upload
  4. Only mobile-side validation runs (server checks pending)
  5. Auto-uploads when internet returns
✅
User experience

The Save button works identically online or offline. Users never need to check connectivity.

  1. Record stays Dirty with error details stored locally
  2. User sees which record failed and exactly why
  3. User opens the record, fixes the issue, re-saves
  4. Auto-retries on the next sync cycle
  5. Other records are not blocked — each syncs independently
