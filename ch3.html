Conditional Visibility — Depends On

Every field (or Section Break) can have a depends_on expression — a condition that determines when that field is visible. When the condition is false, the field hides automatically. Works identically on web and mobile.

// Simple condition — show field only when status equals "Active" doc.status == "Active" // AND condition — show when both conditions are true doc.status == "Active" && doc.program == "Shiksha Plus" // OR condition — show when either condition is true doc.purpose == "Training" || doc.purpose == "Assessment" // Negation — show when field is NOT empty doc.volunteer_id != "" && doc.volunteer_id != null
✅
Key Advantage over mForm

In mForm, skip logic was distributed across parent/child arrays with regex patterns. In Frappe, each field carries one self-contained depends_on expression — readable at a glance.

Mandatory Depends On — Conditional Required Fields

A field can be optional in general but become required when a condition is met, using mandatory_depends_on:

// "Batch Number" is only required when CMS Status = "Running" mandatory_depends_on: doc.cms_status == "Running"
⚠️
Server-side enforcement required

The form definition shows/hides the required indicator, but server-side scripts must explicitly enforce conditional mandatory fields — the server doesn't automatically validate them.

Cascading Dropdowns — Link Filters

Link fields can be filtered based on other fields' values. The most common pattern is geography — selecting a State automatically narrows the District list:

Maharashtra
→ District narrows to
Pune ▾
→ Block narrows to
Haveli ▾
→ Village: Rohini
When State changes → District resets. When District changes → Block resets. Cascades all the way down.
// District field: only show districts where state = selected state filters: { state: ["=", ["state"]] } // Block field: filter by both state and district filters: { state: ["=", ["state"]], district: ["=", ["district"]] }

Auto-fill — Fetch From

When a user selects a record from a Link field, other fields on the form can automatically copy values from that linked record using fetch_from:

Option Masters — Managing Dropdown Choices

1️⃣Pattern 1 — Hardcoded Select (small, stable lists)▼

Choices typed directly into the field definition separated by newlines. Example: Male Female Other. Use only when the list is short (≤5 options) and will never change.

Limitation: Changing options requires editing the DocType definition — needs developer access.

2️⃣Pattern 2 — Link to Option Master DocType (preferred)▼

Options live in a separate DocType (e.g., "Education Level" with records: Illiterate, Primary, Secondary, Graduate). The field is a Link pointing to that DocType.

Benefit: Admins can add/remove options through the UI, no developer needed. Same list can be shared across many forms. Options can have extra metadata (descriptions, ordering).

3️⃣Pattern 3 — Child Table with per-option data▼

When you need to record data about each selected option (score, status, observation), use a Child Table where each row links to an option and has additional fields.

Example: Elementary DVR has a "5E Teaching Method" child table. Each row links to a 5E Option (Engage/Explore/Explain/Elaborate/Evaluate) AND has an observation status field.

Frappe Workflows — Multi-stage Approvals

Workflows define a state machine for a DocType — allowing records to move through stages with different permissions at each stage.

●

Draft

Default state. Field worker fills the form. Can edit freely. Only the creator and their supervisor can see it.

●

Submitted

Field worker submits. Form is now locked — cannot be edited without approval. Coordinator notified.

●

Approved / Rejected

Coordinator reviews and either Approves (finalizes) or Rejects (sends back for correction). Full history tracked in Timeline.

●

Amended (if needed)

If corrections needed after approval, an Amendment creates a new version while preserving the original — full audit trail maintained.
