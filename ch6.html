No form is hardcoded in the app

App Architecture — Thin Shell, Powerful SDK

🏠
App Shell (Project-specific)

Lightweight outer wrapper. Provides: app icon/branding, initial configuration URL, project-specific Dart form handlers, and custom UI overrides. This layer is different per client project.

⚙️
Mobile SDK (Shared Engine)

The reusable engine. Handles: DocType definition parsing, dynamic form rendering as native Flutter widgets, local SQLite storage, sync engine, authentication, offline queue management. Shared across all projects.

The SDK Form Rendering Pipeline

1

Fetch DocType definition

SDK checks local cache. If stale or missing, downloads fresh definition from server. Cached in SQLite with a timestamp.

2

Parse structure: Tabs → Sections → Columns → Fields

SDK reads the ordered field list and builds a tree structure: which fields are in which tab, section, and column.

3

Build Flutter widgets for each field type

Data → TextFormField. Select → DropdownButton. Table → dynamic ListView with Add Row. Attach Image → camera/gallery picker. Each field type maps to a specific Flutter widget.

4

Evaluate conditions — show/hide in real-time

As the user fills fields, depends_on expressions are evaluated. Affected fields show/hide instantly using Flutter's reactive state management.

5

Attach Dart form handlers (on_field_change)

Project-specific handlers are attached to field events. When a field changes, the handler fires — can update other fields, trigger calculations, or show alerts.

Dart Form Handlers — The Mobile Controller

Form Handlers are Dart classes (one per DocType) that provide real-time behavior on the mobile app — the equivalent of Frappe's Web Client Scripts but for Flutter:

// HouseholdSurveyHandler.dart class HouseholdSurveyHandler extends MobileFormHandler { // Fires when any field value changes @override void onFieldChange(String fieldname, dynamic value, FormController ctrl) { switch (fieldname) { case "reading_level": case "writing_level": case "math_level": // Auto-calculate case category from three scores int r = ctrl.getInt("reading_level"); int w = ctrl.getInt("writing_level"); int m = ctrl.getInt("math_level"); ctrl.setValue("case_category", _calculateCase(r, w, m)); break; case "village": // Filter volunteer dropdown when village changes ctrl.setLinkFilter("volunteer_id", { "village": value }); ctrl.clearValue("volunteer_id"); break; } } // Fires before saving — return false to block save @override bool onValidate(FormController ctrl) { if (ctrl.getInt("total_children") == 0) { ctrl.showError("At least one family member must be recorded"); return false; } return true; } }

Offline-First Design

💾
Local SQLite Storage

All form definitions, records, dropdown options, auth tokens, and translations are stored in SQLite on the device. The user never needs to wait for a network request to fill forms.

📶
Transparent Sync

The Save button behaves identically online or offline. The user never needs to check connectivity. If offline, data goes to the local queue and syncs automatically when internet returns.

♾️
No Time Limit

Works offline indefinitely. Thousands of records can accumulate in the local queue. Only storage space limits offline capacity — and form data is lightweight text/numbers.

🔄
Auth Token Rotation

Access token (24h) is refreshed automatically by the refresh token (30 days). Users stay logged in for 30 days of regular use without re-entering credentials.

Record States

ErrorTried to sync but server rejected — validation failed, duplicate key, missing reference.User opens record, sees error message, fixes it, re-saves.
DeletedMarked for deletion locally. Server copy not yet removed.Delete request sent to server on next sync.

Login & Initial Sync Flow

1

Check if mobile service is enabled

App queries the server before showing the login screen. If mobile is disabled by admin, stops here with a message.

2

Authentication

Username/password or Mobile + OTP. On success: receives access token, refresh token, enabled forms list, user permissions, and language preference.

3

Schema Sync — download form definitions

Downloads all DocType definitions for enabled forms. Each definition includes fields, layout, conditions, naming rules, and permissions.

4

Data Sync (Pull) — download existing records

Downloads records the user has permission to see. Also downloads all dropdown option master records and translations.

5

Ready — go offline

All data needed for offline work is now on the device. Internet can be switched off. The user can fill any form immediately.
