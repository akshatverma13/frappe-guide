Three Types of Sync

TypeDirectionWhat TravelsWhen
Schema SyncServer → AppForm definitions (fields, layout, conditions)App startup, after form changes
Push SyncApp → ServerFilled records (new, modified, deleted)After save, startup, manual button
Pull SyncServer → AppOther users' records, Desk edits, new master dataApp startup, after push, manual

Push Sync — Dependency Ordering

Records don't upload in random order. They follow dependency sequence so all references are resolved on the server first.

Level 0 — First: Geography masters (State, District, Block, Village, Hamlet)
Level 1 — Next: People masters (Employees, Volunteers, Households)
Level 2 — Then: Transaction forms (DVRs, Attendance, Surveys)
Level 3 — Last: Forms that reference other transactions
⚠️
Why order matters

A Visit Report references a Volunteer. If the Volunteer hasn't uploaded yet when the Visit Report tries to sync, the server rejects it. Ordering prevents this.

Pull Sync — Delta Approach

The app only asks for what's changed since its last sync — not the entire database every time.

The app asks the server:
"Give me all Household Survey records modified after 2024-01-15 10:30:00"

Only changed records come down — fast even with thousands of records.

Schema Sync — Automatic Form Updates

1

Admin changes a form on the server

Adds a field, changes a label, adds a new dropdown option.

2

Server updates the "last modified" timestamp

Tracks exactly when each form definition changed.

3

App compares timestamps on next sync

"My cached version is from Jan 15. Server says Jan 20. I need to update."

4

App downloads the fresh definition

Users see the updated form next time they open it. No Play Store update ever required.
