Data Flow & Sync

The complete data lifecycle. Three sync types, dependency ordering, delta sync, and error handling.

Three Types of Sync

TypeDirectionWhat TravelsWhen
Schema SyncServer → AppForm definitions (fields, layout, conditions)App startup, after form changes
Push SyncApp → ServerFilled records (new, modified, deleted)After save, startup, manual button
Pull SyncServer → AppOther users' records, Desk edits, new master dataApp startup, after push, manual

Push Sync — Dependency Ordering

Records upload in a specific sequence so all references resolve on the server before they are needed.

Level 0 — First  ·  Geography masters (State, District, Block, Village, Hamlet)
Level 1 — Next  ·  People masters (Employees, Volunteers, Households)
Level 2 — Then  ·  Transaction forms (DVRs, Attendance, Surveys)
Level 3 — Last  ·  Forms that reference other transactions
⚠️
Why order matters

A Visit Report references a Volunteer. If the Volunteer hasn't uploaded yet, the server rejects the Visit Report. Dependency ordering prevents this.

Pull Sync — Delta Approach

The app only asks for what has changed since its last sync — not the entire database every time.

App asks the server:
"Give me all Household Survey records modified after 2024-01-15 10:30:00"

Only changed records come down — fast even with thousands of records in the system.

Schema Sync — Automatic Form Updates

1

Admin changes a form on the server

Adds a field, changes a label, or adds a new dropdown option.

2

Server records a "last modified" timestamp

Tracks exactly when each form definition last changed.

3

App compares timestamps on next sync

"My cached version is from Jan 15. Server says Jan 20. Download the fresh version."

4

Updated form appears automatically

No Play Store update ever required — the app updates forms in the background.
