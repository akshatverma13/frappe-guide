Push Sync — Dependency Ordering

Records don't upload randomly — they follow a strict dependency sequence to ensure all referenced records exist on the server before forms that link to them:

0
Geography Masters
State, District, Block, Gram Panchayat, Village, Hamlet · No dependencies
1
People Masters
Employees, Volunteers, Households · Depend on Level 0
2
Transaction Forms
DVRs, Attendance, Surveys, MIS · Depend on Levels 0 + 1
3
Referencing Transactions
Assessments, BME · Depend on Levels 0 + 1 + 2
⚠️
Why ordering matters

A Visit Report links to a Volunteer. If the Volunteer hasn't uploaded yet, the server rejects the Visit Report with a "linked record does not exist" error. Ordering prevents this entirely.

Delta Sync — Efficient Data Download

Pull Sync uses a "delta" approach — the app only requests records that have changed since its last sync:

// App sends this query on Pull Sync: GET /api/resource/Household Survey ?filters=[["modified", ">", "2024-04-10 14:30:00"]] &fields=["name","modified","village","head_of_household",...] // Server returns ONLY records modified since that timestamp // Not the entire database — just the delta
✅
Result

Even with 50,000 records in the system, if only 12 changed since last sync, only 12 come down. Syncs are fast even on slow mobile data connections.

Permission Levels

🌐DocType-Level Permissions▼

Controls what a role can do with a DocType: Read, Create, Write, Delete, Submit, Cancel, Amend, Report, Export, Import, Share. Set in the DocType's Permissions table.

🔏User Permissions — "Own Records Only"▼

User Permissions restrict a user to only see/edit records where a specific link field matches their value. Example: "Field Coordinator Kavita can only see DVRs where created_by = Kavita." Set in User Permissions DocType.

🔒Field-Level Permissions (Permlevel)▼

Each field has a "Permlevel" (0–9). Users need the corresponding permlevel on their role to see/edit that field. Permlevel 0 = everyone. Permlevel 1+ = only roles with that specific permission level. Useful for hiding sensitive fields (salary, phone numbers) from certain roles.

Fixtures — Pre-loaded Data

Fixtures are data records bundled with a Frappe App that are automatically installed (or updated) when the app is installed. They ensure every fresh installation has the same base data:

Fixture TypeExampleWhy Use Fixtures
Option MastersEducation Level records: Illiterate, Primary, Secondary...Same options on every installation — no manual data entry
Geography DataStates, Districts, Blocks (seed data)Pre-populate geography without manual imports
Role ProfilesMobile User, Field Coordinator, State CoordinatorConsistent roles across environments (dev, staging, prod)
Custom FieldsExtra fields added to standard DocTypesEnsure custom fields are reproduced in every install
Notification RulesEmail alerts on DVR submissionSystem configuration deployed automatically

Error Handling in Sync

Error ScenarioWhat HappensHow to Recover
No internet for daysRecords accumulate as Dirty. Nothing lost.Auto-batch-upload when internet returns.
Server rejects a recordRecord stays Dirty. Error text stored locally.User opens record, sees error, fixes, re-saves.
Partial sync failureSuccessful records become Clean. Failed stay Dirty.Next sync retries only the failed records.
Two users edit the same recordLast-write-wins (by server timestamp).Rare in practice — workers cover separate areas.
Referenced record missingUpload rejected: "linked record does not exist".Check dependency ordering — upload the referenced record first.
Auth token expiredApp auto-refreshes using refresh token (30-day).If refresh token also expired — user re-logs in once.
